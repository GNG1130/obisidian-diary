- 初步透過建立的KG來生成題目，並且使用了 Naive, Graph, Hippo 三種不一樣的 RAG 技術進行評估
	- 主要分為以下五個 subgraph 型式 : path, star, constraint, hybrid, comparison
	- ![[anchorbench_topology_chinese.svg]]

> [!example]- PATH
> **用途：** 測試 sequential reasoning 和 bridge entity tracing。問題需要先找到中介概念，再從中介概念找到答案，代表最基本的多跳推理能力。
> **遭遇的問題：** 問題品質最穩定，但答案有時過短（只有一個名詞），不夠完整。在所有 topology 中對所有 RAG 方法都是最困難的（Pilot: HippoRAG LLM Judge 僅 6.7%），顯示線性推理路徑仍是現有 RAG 的核心瓶頸。

> [!example]- Star
> **用途：** 測試 attribute aggregation 和 central entity identification。問題需要整合同一中心概念的多個屬性，最接近人類真實提問方式（「請描述 X 的各個面向」）。
**遭遇的問題：** 在學類限制下候選池最少（5% 通過率），因為同一中心 entity 要在同學類內連出三條不同關係到三個不同 tail 的條件很嚴苛。Hub node（如「滿意度」、「學生」）作為中心 entity 時會吸引跨主題的鄰居，導致問題語意拼湊感強，需要過濾。

> [!example]- Comparison
> **用途：** 測試 dual-branch retrieval 和 comparative reasoning。問題需要從兩個不同來源各取一個值並進行比較，是最能區分 chunk-based 和 graph-based RAG 能力的 topology（Pilot: Graph RAG 在此 topology 進步最顯著 +25%）。
> **遭遇的問題：** 在 pilot 規模（28,000 篇）下最難生成。即使放寬 similarity 閾值至 0.25–0.95 並加入學類限制，LLM 仍對大多數候選回傳 skip，因為「相同 relation type」的配對條件（如兩篇論文都有「導致」關係）在語意上不保證可比較性。全量資料跑完後預計大幅改善。

> [!example]- Constraint
> **用途：** 測試 multi-condition filtering 和 constraint satisfaction。問題描述兩個條件，要求找出同時滿足的答案，模擬真實世界「尋找符合特定需求的方案」的提問情境。
**遭遇的問題：** 初版 LLM 過濾時，LLM 誤將 constraint 問題中「條件出現在問題裡」判定為 shortcut，導致大量誤刪。修正為客觀指標（single chunk sufficiency）後問題解決。另外答案容易過短（只有一個名詞），需要在 prompt 中明確要求完整陳述。

> [!example]- Hybrid
> **用途：** 測試綜合推理能力，結合 path 的路徑追蹤和 star 的屬性聚合。問題需要先找到中介概念，再整合該中介概念的多個面向，是五種 topology 中最具挑戰性的。
> **遭遇的問題：** Hub node 問題最嚴重，中介節點（bridge entity）容易是高度數節點（如「台灣」、「滿意度」），導致兩個分支來自完全不相關的領域，問題出現語意跳躍。加入 source_chunks 數量上限（≤ 6）和 field-level 軟性限制後改善。問題品質反而比 comparison 和 constraint 更自然，但 supporting chunks 來源較模糊，人工審查時不易直觀判斷。
